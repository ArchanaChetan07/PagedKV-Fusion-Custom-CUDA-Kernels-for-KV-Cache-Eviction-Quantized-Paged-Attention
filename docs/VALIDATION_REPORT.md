## 3. Quantization quality (measured, CPU, reference math)

From `results/quant_bench_cpu_sections.json`, generated by
`benchmarks/bench_quant_attention.py::measure_quality` (seed=0, 8 sequences,
max seq len 512, per-(block,head) symmetric INT8):

| KV distribution | RMSE | Max abs error | Output std (fp32 baseline) |
|---|---|---|---|
| Gaussian, unit variance | 0.00239 | 0.0161 | 0.233 |
| Heavy-tailed (0.1% of K entries scaled ×10) | 0.312 | 3.57 | 7.02 |

**Reading this honestly:** on well-behaved (roughly Gaussian) KV values —
which is close to what trained transformer activations typically look like
— per-block INT8 keeps RMSE under 1% of output scale. On a distribution
with rare large outliers, RMSE jumps to ~4.4% of output scale and the max
error becomes comparable to the signal itself. **This is the known failure
mode of per-block symmetric quantization: a single outlier blows up the
block's scale factor and crushes every other value in that block toward
zero.** This matches the risk flagged in the project plan ("Quantization
hurts output quality unacceptably") and is reported here as a real,
measured tradeoff rather than glossed over.

Implication for v1 scope: per-(block, head) granularity is likely
insufficient if real model KV caches have channel-wise outliers (a
documented phenomenon in quantization literature for activations). A
follow-up experiment worth running before shipping this quantization scheme
broadly: repeat this measurement on **actual model KV cache dumps** (not
synthetic Gaussian/outlier data) — outlier structure in real activations
may be more or less regular than this synthetic stress test assumes,
which changes whether per-block scaling is adequate or whether per-channel
scales / mixed precision would be required.

*Not yet measured: downstream task accuracy / perplexity delta (Deliverable
4 in the project plan). This requires running a real model, which requires
a GPU. See §6 for what's blocked and what's not.*

### 3a. Downstream-decision proxy (measured, CPU, synthetic — not real perplexity)

`benchmarks/bench_downstream_proxy.py` (`results/downstream_proxy_cpu.json`)
constructs a synthetic nearest-prototype classification task — the query
matches one of 20 "concept" vectors, KV rows are a mix of the true concept
and distractors, and the attention output is classified by nearest
prototype. This is **not real perplexity** (that needs an actual model +
GPU, see §6), but it targets the same failure mode: does quantization flip
a downstream decision, not just perturb a raw output vector.

| seq_len | noise_std | FP32 acc. | INT8 acc. | FP32-vs-INT8 disagreement |
|---|---|---|---|---|
| 16 | 0.05 (easy) | 100.0% | 100.0% | 0.0% |
| 16 | 0.30 | 100.0% | 100.0% | 0.0% |
| 16 | 0.80 (hard) | 62.9% | 62.7% | 0.55% |
| 64 | 0.05 (easy) | 100.0% | 100.0% | 0.0% |
| 64 | 0.30 | 100.0% | 100.0% | 0.0% |
| 64 | 0.80 (hard) | 97.0% | 97.0% | 0.10% |

The number that matters most here is the **disagreement rate**, not raw
accuracy — it isolates decisions quantization actually changed, separate
from how hard the synthetic task is. It stays under 0.6% even in the
hardest, noisiest setting, and INT8 accuracy never drops meaningfully below
FP32. Read this as: *on this proxy, per-block INT8 rarely changes what the
model would have decided* — a positive signal, though one that should be
re-checked against real model perplexity once GPU access is available,
since real activation geometry could differ from this synthetic setup.

### 3b. End-to-end pipeline composition (measured, CPU, reference path)

Every other test/benchmark in this repo exercises one component. Nothing
above proves eviction scoring, quantization, and paged attention actually
*compose* into one working system — a unit-test suite passing per-function
doesn't guarantee that. `scripts/run_end_to_end_demo.py` does prove this:
it runs a simulated scheduler step — score all blocks, evict the bottom
15%, quantize survivors, run a decode-step attention pass — as one
executed pipeline, on the same reference math the tests validate.

Example run (`--num-seqs 48 --max-seq-len 768`, reference backend):

```
backend in use: reference
[1/4] built paged KV state: 1155 physical blocks, 48 sequences in 315.55 ms (numpy RNG, not pipeline logic)
[2/4] scored 1155 blocks, selected 173 for eviction (15.0%) in 0.71 ms
       43/48 sequences reference an evicted block (would trigger a refill in a real scheduler)
[3/4] quantized 1155 blocks to INT8 in 141.80 ms — 72.2 MiB -> 18.1 MiB, 74.9% savings vs fp32
[4/4] ran paged attention decode step for 48 sequences in 70.38 ms; output finite: True
pipeline OK end-to-end. total wall time: 528.44 ms
```

Note this correctly surfaces a real cross-component interaction a pure
unit test never would: evicting 15% of physical blocks affects **43 of 48
sequences** (nearly all of them), because block tables are shared/striped
across the eviction pool in this synthetic setup — a concrete illustration
of why eviction policy and scheduler refill logic can't be designed in
isolation from each other. (The 74.9% figure here is INT8-vs-fp32, not the
~50% INT8-vs-fp16 figure in §4 — vLLM stores KV in fp16, so §4's number is
the one that matters for real deployment; this script's fp32 comparison is
just what the synthetic setup generates data in.)

This is reference-path timing, not a GPU throughput number — see §6.

## 4. Memory footprint (exact arithmetic, no GPU required)

From `results/quant_bench_cpu_sections.json::memory`:

| Config | FP16 KV cache | INT8 KV cache (incl. fp32 scales) | Savings |
|---|---|---|---|
| 4,096 blocks, 8 KV heads, head_dim 128 (≈7B-class) | 256.0 MiB | 128.25 MiB | 49.9% |
| 16,384 blocks, 8 KV heads, head_dim 128 | 1024.0 MiB | 513.0 MiB | 49.9% |
| 16,384 blocks, 40 KV heads, head_dim 128 (≈13B MHA) | 5120.0 MiB | 2565.0 MiB | 49.9% |

This hits the project's stated ~50% target — it's exact arithmetic
(int8 = 1 byte vs fp16 = 2 bytes, plus a small fp32-scale overhead), not a
measurement that depends on hardware, so it's safe to state with full
confidence independent of GPU availability. The overhead of storing
per-(block, head) fp32 scales is what keeps savings just under the naive
2× (0.1% overhead at head_dim=128, more visible at smaller head_dim).

### 5b. CUDA fused kernel (GPU — NVIDIA T1000, CUDA 12.5)

From `results/eviction_bench_gpu_sections.json` (200 timed iterations):

| num_blocks | impl | p50 latency | p90 latency | Speedup vs host (p50) |
|---|---|---|---|---|
| 1,024 | `cuda_fused` | 81.0 µs | 82.2 µs | 1.1× |
| 16,384 | `cuda_fused` | 562.0 µs | 590.5 µs | **3.0×** |
| 16,384 | `host_numpy` | 1707.8 µs | 1956.4 µs | (baseline) |

The crossover where the fused kernel wins clearly is at scale (16K blocks).
At 1K blocks, launch overhead dominates. On a serving box with device→host
attention-stats copies, the host baseline would be worse than measured here.

### 7b. INT8 paged-attention throughput

From `results/quant_bench_gpu_sections.json` (`bench_quant_attention.py`,
10 timed iterations per config; SDPA iterations adaptively capped when probe
> 500 ms):

| num_seqs | seq_len | INT8 kernel p50 (ms) | fp16 SDPA gathered p50 (ms) | Ratio |
|---|---|---|---|---|
| 8 | 512 | 1.76 | 9.01 | 5.1× |
| 32 | 1024 | 3.34 | 70.00 | 21× |
| 64 | 2048 | 14.18 | 27,320 | 1,927× |

**Reading this honestly:** the INT8 paged kernel is consistently faster than
the gathered fp16 SDPA baseline on this hardware. The SDPA baseline degrades
catastrophically at large batch×sequence because consumer GPUs lack flash-attn
for this layout — it is an upper-bound comparison, not a production vLLM path.
The INT8 kernel numbers are the meaningful signal: sub-2 ms at modest decode
batch, ~14 ms at 64 sequences × 2048 context on a T1000.
